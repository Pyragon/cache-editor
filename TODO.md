## Dump change tracking (manifest migration)

New system (2026-07-16), replacing the per-dumper `.metadata` mtime files: `DumpManifest` (cryogen, `com.cryo.cache`) records size + mtime + content CRC32 per entry, with size+mtime as a no-read fast path and the hash as confirmation — so touches, copies, and re-dumps that produce identical content are silently re-baselined instead of repacked, and (new capability) multi-file entries hash ALL their files, so e.g. hand-replacing a midi_instrument's `sound.ogg` alone now correctly triggers a repack (the old `.metadata` only ever tracked `data.json`'s mtime — a latent bug). `ManifestBackedDefinition` implements the `getActions()`/`rebuildMetadata()` lifecycle exactly once; a dumper now only declares its dump dir, its entry→files enumeration (helpers for the three shapes: flat `<id>.json`, folder-per-id with named files, two-level `<archive>/<file>.json`), and its add-edit/remove action builders. Legacy `.metadata` files are deleted automatically the first time a manifest is written; a missing manifest packs everything (preserving old first-pack behavior). Full lifecycle verified against a temp copy of the real dump by `DumpManifestLifecycleTest` (13/13 checks: baseline, clean-tree no-op, touch no-op with persisted re-baseline, single edit → single action, rebuild, deletion → remove action, cursors' fixed path, legacy cleanup, no-manifest full pack).

**Migration complete** (2026-07-16): all 42 dumpers with a real `.metadata` pattern are now on `ManifestBackedDefinition` — verified end-to-end by `DumpManifestLifecycleTest` (19/19 checks against temp copies of the real dump, covering the full lifecycle plus multi-file entries: a PNG-only sprite edit repacks, a `song.bin`-only music edit correctly doesn't). The one former exception, `BASDefinitions`, was migrated in the 2026-07-19 bas pass (see the BAS section) when the entry was dumped for the first time.

Notable decisions/finds from the migration:
- **`buildAddEdit`/`buildRemove` return lists**, because one map region packs two archives (terrain "m" + XTEA "l").
- **`EntityDefaults` + `EquipmentDefaults` merged** into one `DefaultsDefinitions` (they shared one folder and thus one tracking file); both stay as decode/encode data classes.
- **`quests`/`map_areas` key by the parsed leading id** of their `"<id> - <Name>.json"` filenames, so renaming an entry reads as a content change on a stable key rather than a delete + re-add racing against the same archive id.
- **Entry file rules for folder-per-id dumps**: sprites = json + every frame PNG (both repack); models = `model.dat` only; music = `song.mid` only (`song.bin` is a reference artifact); sound_effects/textures = json only (the `.wav`/`.png` siblings are generated previews).
- **Two more silently-broken repack paths found and fixed by the migration** (beyond cursors'): `ModelDefinitions` (compared the *directory's* mtime, targeted the singleton's own id through a doubled path, and `rebuildMetadata` skipped directories so it always wrote an empty file) and `FontMetrics` (`addAllFiles` targeted the singleton's `id` field — always 0 — so every repacked font would have hit archive 0). Plus two latent multi-file bugs: sprites' old tracker only watched PNGs (a JSON-only edit never repacked) and midi_instruments' only watched `data.json` (a hand-swapped `sound.ogg` never repacked).
- Pre-existing oddity preserved-not-fixed: `QuestDefinitions`' two old code paths disagreed on `archiveId(id)` vs `getId()`; the maintained `getActions` semantics were kept.

## Sound / Music

All four sound/music indices got real work this pass (2026-07-15):

**`sound_effects`** — cryogen's `SoundEffect`/`Instrument`/`Envelope`/`Filter` already fully decoded the format and rendered real WAV previews via a genuine additive-synthesis engine (oscillators, envelopes, an optional biquad filter, echo); `encode()` was a stub returning `null`. Added the full inverse chain. Round-trip testing (`SoundEffectRoundTripTest`) found and fixed three real bugs, none related to this pass's own changes:
- The oscillator-array terminator was written conditionally on "array not full", but `decode()` always reads one more terminating value after the real entries even when all 5 slots are used (the read happens before the array-bounds check that would actually crash) — always write it now.
- `Filter`'s "migrated" bitmask (which end-of-envelope values are explicitly distinct vs. copied) was being re-inferred from decoded state, but real cache data sometimes sets a migrated bit even when the value doesn't change — now stored as a raw field from `decode()` and used as-is.
- `encode()` returned `OutputStream.getBuffer()`, which is the stream's raw internal array (starts at 16 bytes, only grows, never shrinks) — any effect encoding to ≤16 bytes got silent trailing garbage. Switched to `toByteArray()`, which trims to the actual written length.

Also marked `Envelope`'s runtime synthesis-scratch fields (`critical`/`phaseIndex`/`step`/`amplitude`/`ticks`) `transient` — `dumpFiles()` renders the WAV *before* serializing to JSON, so without this the dump captured whatever synthesis state was left over from rendering, not real decoded data.

Result: **100% byte-identical round-trip, 10,084/10,084 components.** Editor: full field editor (timing, 5 oscillators, 6 envelope pairs, filter) with drawn envelope curves, plus **live in-browser re-synthesis** — the synth engine (`Instrument.synthesize()`/`Envelope.step()`/`Filter.compute()`/`SoundEffect.mix()`) is ported to TypeScript (`src/loaders/soundSynth.ts`), verified **sample-exact against all 10,084 dumped WAVs** (the port replicates Java int wrap/saturating casts, 32-bit float filter math, and the JDK `Random(0)` noise table), so edits play instantly via WebAudio instead of the preview going stale until a re-dump.

**`midi_instruments`** — despite the name, this is Ogg Vorbis sample audio (not General MIDI), the sample bank the music sequencer's voices play. `dumpFiles()` already reconstructed real playable `.ogg` files (via the `gagravarr` Vorbis-Java library, borrowing header packets from a template `data/example.ogg`); `encode()` was a stub, and `fromOGGFile()` was an unfinished debug method that printed packet info and returned `null`. Also found and fixed a real bug: `getActions()`/`addAllFiles()` built every `AddEditAction` using `this.id` — the *instance's own* id, always `-1` since the registry-level `Definition` singleton is constructed as `new MidiInstrument(-1)` — instead of the id of the file actually being processed, so every repacked instrument would have silently targeted archive `-1`. Fixed by deriving the id from each file's own directory name. Completed the packet-extraction path (`readOGGPackets`, skips the 3 standard Vorbis header packets, takes everything after as audio data — works for both an untouched re-save and a completely different uploaded `.ogg`, since any standard Vorbis file has the same 3-header-packet structure). Result: **100% byte-identical round-trip, 16,824/16,824.** Editor: metadata fields (sample rate, loop points), real `.ogg` playback, "replace audio" upload.

**`music` / `music2`** — no prior code existed for either at all. Reverse-engineered by tracing the client's actual playback path (`SongReference` → `Node_Sub7` → `Class80`) in darkan-game-client: these archives are **not** literal Standard MIDI Files, they're a Jagex-specific compact re-encoding of one — MIDI events are split into several independently delta-coded parallel byte streams grouped by event type (and, for Control Change events, further split per controller number), which `Node_Sub7`'s constructor reassembles into a genuine spec-compliant `.mid` file. Ported that constructor faithfully to `MusicTrackDecompressor.java` (variable names intentionally kept close to the obfuscated source rather than renamed under time pressure — a byte-for-byte structural port where one wrong offset silently corrupts output, verifiable side-by-side with the reference was judged safer). Found and fixed one real bug along the way: the compact format's "skip repeated status byte" optimization was being applied to META events (tempo, end-of-track) the same way it's applied to real channel messages, but MIDI running status never applies to meta/sysex events — standard-compliant parsers require every meta event to carry its own `0xff` prefix. The bug only manifests when a meta event's low nibble happens to numerically collide with the previous event's (e.g. `SET_TEMPO`'s sentinel `23 & 0xf == 7`, the same nibble `END_OF_TRACK` uses) — which happens to be *every* song's track 0 (a dedicated tempo-map track: tempo event immediately followed by end-of-track). The game's own lenient re-parser (`Class80`) tolerates the omission; Java's standards-compliant `javax.sound.midi` does not. Fixed by always writing `0xff` for meta events regardless of the "changed status" check (and fixing the matching size-estimation pass to match). Result: **100% valid output, 1,137/1,137 (`music`) and 525/525 (`music2`)**, verified by successfully parsing every decompressed file with `javax.sound.midi.MidiSystem.getSequence()`.

**The inverse (compression) direction is now implemented too** (`MusicTrackCompressor.java`), completing the repack path. There was a prior, independent, unfinished attempt at exactly this in `MIDIEncoder.java` (uses `javax.sound.midi` to read a `.mid` and re-emit the same per-event-type stream grouping `MusicTrackDecompressor` decodes — matching opcode numbering and comments like *"jagex works with offset from the last one because this is usually 0 and gives better compression rates"* confirmed real prior reverse-engineering effort), but it was a `main()` method with hardcoded filenames, never integrated into cryogen's `Definition`/`CacheAction` pipeline, and never validated. Cross-checking its per-event-type stream ordering against `MusicTrackDecompressor`'s independently-derived cursor layout found it matched exactly — strong evidence it was reverse-engineered correctly — except for one real bug: the running channel-XOR accumulator (`prevChannel`) was reset to 0 at the start of every track, but the decompressor's matching accumulator persists across an entire song's tracks. Fixed, wrapped into a real `compress(byte[] midiBytes): byte[]` method, and validated against all 1,662 real songs across both indices (decompress → compress → decompress): **100% functionally identical decoded MIDI content** (only 18/1,662 are byte-identical to Jagex's own original encoder output, but every single one decodes back to the exact same music — the accepted round-trip bar).

Dumper (`MusicDefinitions.java`) writes both `song.bin` (raw cache bytes, reference/inspection only) and `song.mid` (the decompressed standard MIDI file, now the *editable* source of truth — `getActions()` recompresses whatever's in `song.mid`, not `song.bin`) per song. Editor: real playable download link (browsers don't play raw `.mid` without an attached synth, so this is a link, not an `<audio>` tag), a "Replace with edited .mid…" upload, and a Save bar wired through to a real repack.

**`sound_effects_midi`** — decoded and repacked in a follow-up pass (2026-07-16). The name is misleading a third time: it's neither the `sound_effects` synthesis format nor the `music` compact-MIDI format. Traced via the client (`Class42_Sub1.method14563` → `Static.method2084` → `Node_Sub15_Sub2.method15182` → `PlaySoundJingleCutsceneAction.method14676` → `Node_Sub14`'s byte[] constructor): each archive is a SoundFont/DLS-style instrument bank keyed by MIDI program-change number — a 128-note keymap where each note points at a "zone" (`Class110`) carrying an amplitude envelope, a continuous decay rate, and a vibrato LFO, plus per-note choke/exclusive-class group, pan, volume, and tuning. No raw audio lives here at all — each note's sample code cross-references either `sound_effects` or `midi_instruments` (resolved by the client's `Node_Sub14.method12216`).

Ported faithfully to `SoundEffectMidi.java` (same "keep obfuscated variable names, verify byte-for-byte against the reference" approach as `MusicTrackDecompressor`), with one deliberate deviation: the client bakes two keyboard-wide curves (a volume curve and a pan curve) directly into the per-note volume/pan tables at decode time — lossy and not cleanly invertible — so this port stores the pre-bake per-note values plus the two raw curves separately, letting `encode()` reconstruct the exact original bytes without inverting that baking step. Writing the encoder surfaced two real format subtleties (not bugs in the client, just easy to get wrong porting blind): the earlier reverse-engineering pass assumed each run-length table's reserved value region had one unused trailing byte, but it's actually fully consumed — the *last* group in each of the four run-length tables (choke group, pan, zone assignment, and the shared sample-code/volume grouping) is implicit/open-ended (no explicit length is written for it, matching the decoder's own "-1 forever" catch-all once the run-length array is exhausted). Missing this initially produced a consistent +6-byte/archive mismatch across every test file. Result: **100% byte-identical round-trip, 246/246 non-empty archives**, verified against the real cache with `SoundEffectMidiRoundTripTest`.

Editor (`src/loaders/sound_effects_midi.ts`, `SoundEffectMidiViewer.tsx`): global gain and per-zone fields (envelopes as editable breakpoint tables, decay/vibrato rates) are fully editable; the 128-note keymap is shown read-only, grouped into contiguous note ranges with a resolved link to the referenced `sound_effects`/`midi_instruments` entry — editing individual notes across six 128-entry parallel arrays didn't have a clean enough UI to justify building this pass. **Not yet tested in a real browser session** (same File System Access API limitation as other recent viewers — no headless-browser tool available here); typecheck, lint, and production build all pass, and the underlying dumper/decoder is verified byte-identical against live cache data independent of the browser-side editor.

## Interfaces

The `interfaces` entry now has a real dumper/encoder/viewer (2026-07-15). cryogen `IComponentDefinitions` dumps `interfaces/<interfaceId>/<componentId>.json` — one folder per interface, one JSON file per component (no `components/` subfolder, no per-interface screenshot; both the old nested-with-screenshot layout and an intermediate flat-array-per-interface layout were tried and replaced before landing here). Field names were cross-referenced against darkan-bot-refactor's `Component.kt` and renamed (~35 fields); ~15 fields remain unidentified obfuscated names (`anObjectArray1413` etc. — CS2 script hooks whose purpose wasn't cross-referenced yet).

Writing this dumper for the first time surfaced three real, previously-invisible bugs in the (pre-existing) `encode()` — nothing to do with the rename, all confirmed via a byte-identical round-trip test (`IComponentRoundTripTest`, package `com.cryo.cache.loaders.interfaces`, left in the repo per the existing `*RoundTripTest` convention) against all 1,323 interfaces / 65,259 components:
- **MODEL components**: the transform/origin/orthogonal/priority-render flag byte was computed but never written — every MODEL component's encode was 1 byte short.
- **Menu option cursors**: the packed byte combining option count + cursor mask only ever wrote the count (cursor mask nibble was always 0 on encode, so the whole cursor-mask branch was dead), *and* the cursor value itself was read from the wrong array index (`length-2` instead of `length-1`).
- **Cycle-step / key-trigger entries**: `decode()` reused a single field as both the persistent value and the loop's scratch variable, so by the time decode finished the field held the loop's terminal value (usually 0) instead of the real one — encode's `if (hash != 0)` guard almost never fired, silently dropping these entries. Also separately, encode wrote each entry as a 4-byte int instead of the real ~2-byte packed format. Both fixed; `decode()` now uses a local loop variable and doesn't clobber persistent state.

The `decode()` text mutation that silently rewrote "RuneScape" → "Darkan" in TEXT components (a leftover from some earlier rebranding experiment — it would have permanently overwritten the live cache's original strings on any save) has been removed entirely; `text` now round-trips as dumped.

Two more root causes were found and fixed chasing full byte-identical fidelity:
- **`InputStream.readString()`/`OutputStream.writeString()` had no real charset table at all.** `readString()` sign-extended each raw signed Java `byte` straight into a `char`, corrupting every codepoint ≥0x80 into the wrong Unicode range. `writeString()` used `String.getBytes()` with no charset argument — the JVM platform-default (confirmed `windows-1252` on this JDK17/Windows setup, not UTF-8), non-deterministic and wrong regardless. RS text is actually Windows-1252 with a real 32-entry special-character table for bytes 0x80-0x9f (curly quotes, em-dash, euro sign, etc.). Ported the exact table from darkan-game-client (`Class490.aCharArray5766` for decode, `Index.method5693` for encode) into both classes — this is shared low-level infra, so it should also fix special characters in any other entry's text, not just interfaces.
- **The JSON dump writer itself wasn't UTF-8.** `dumpFiles()`/`fromFile()` used `new FileWriter(file)`/`new FileReader(file)` with no charset, defaulting to the same platform `windows-1252` — any non-ASCII character was written as a single raw byte, invalid UTF-8, which every reader (the browser, Node, this editor) rendered as `�`. Fixed by writing/reading with explicit UTF-8 in `IComponentDefinitions`. **This exact `new FileWriter`/`new FileReader` pattern (no charset argument) exists in ~62 other files across cryogen** — only this one was touched; any other entry with non-ASCII text in its dump likely has the same silent corruption and would need the same fix.
- **One genuinely inert byte, preserved rather than dropped.** One real component (interface 1320/component 1) has an extra bit (0x20) set in its MODEL flag byte that darkan's actual client decode never reads (confirmed against source — it only tests bits 0x1/0x2/0x4/0x8). Rather than silently losing it on re-encode, it's now captured (`modelFlagUnusedBits`) and written back verbatim. A second, more general safety net was added alongside it: any bytes at all that `decode()` doesn't consume for a given component are captured as `trailingUnreadBytes` and appended back on `encode()`, so any other not-yet-understood trailing data (should any exist) round-trips instead of being silently dropped.
